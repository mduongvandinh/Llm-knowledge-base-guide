01 Tại sao bạn cần một Knowledge Base?

Bạn đọc hàng trăm bài viết mỗi năm. Bạn bookmark, save, highlight. Rồi khi cần lại… bạn Google lại từ đầu. Kiến thức không tích lũy — nó rơi vào hố đen.

Đây là vấn đề mà mọi người gặp: thu thập thì dễ, tổ chức thì khó, duy trì thì gần như không thể.

Hình dung thế này

Cách cũ

Bạn có 500 bookmarks, 200 bài đã save, 50 ghi chú rải rác. Khi cần tìm gì đó, bạn search lại từ đầu. Mỗi lần hỏi, bạn phải tự ghép mảnh thông tin lại.

→
Cách mới

Bạn có một "cuốn bách khoa" do AI viết và duy trì cho bạn. Mọi thứ đã được tổng hợp, liên kết, cập nhật. Bạn chỉ cần hỏi và đọc.

LLM Knowledge Base giải quyết vấn đề này bằng một ý tưởng đơn giản: để AI làm phần việc nhàm chán — tóm tắt, phân loại, liên kết, cập nhật — còn bạn chỉ tập trung vào việc đọc, suy nghĩ, và đặt câu hỏi hay.

02 Ý tưởng cốt lõi

Một câu tóm tắt
Bạn thu thập tài liệu thô → AI "biên dịch" thành một wiki có cấu trúc → bạn hỏi-đáp trên wiki đó → mỗi câu hỏi làm wiki giàu hơn → lặp lại.

Khác gì so với ChatGPT / NotebookLM / RAG?

Hầu hết mọi người dùng AI với tài liệu theo kiểu RAG (Retrieval-Augmented Generation): upload file lên, hỏi câu hỏi, AI tìm đoạn liên quan rồi trả lời. Mỗi lần hỏi, AI phải "khám phá lại" kiến thức từ đầu.

So sánh dễ hiểu

RAG / ChatGPT

Giống như bạn có một thư ký đọc nhanh. Mỗi lần bạn hỏi, thư ký lục tủ hồ sơ, tìm đoạn liên quan, rồi trả lời. Nhưng thư ký không nhớ gì từ lần trước. Hỏi câu tương tự ngày mai → lục lại từ đầu.

→
LLM Wiki

Giống như bạn có một chuyên gia viết sách cho bạn. Chuyên gia đọc từng tài liệu, viết thành chương sách, liên kết các chương lại. Khi bạn hỏi, chuyên gia mở đúng chương cần thiết. Mỗi tài liệu mới → sách được cập nhật.

Điểm khác biệt quan trọng nhất: wiki là một sản phẩm tích lũy. Nó không biến mất sau mỗi cuộc hội thoại. Nó ngày càng giàu hơn, liên kết hơn, hữu ích hơn — giống như tiền gửi tiết kiệm với lãi kép.

03 Kiến trúc — 3 tầng đơn giản

Toàn bộ hệ thống chỉ có 3 thành phần. Không cần database, không cần server, không cần cloud. Tất cả là file markdown trên máy tính của bạn.

Tầng 1: Raw Sources — Tài liệu gốc

Đây là "nguyên liệu" — bài viết, paper, code, ghi chú. Bạn chỉ thêm vào, không bao giờ sửa. AI đọc từ đây nhưng không chạm vào. Đây là nguồn sự thật (source of truth) của bạn.

raw/
├── articles/     ← Bài blog, tutorial clip từ web
├── papers/       ← Papers, tài liệu chuyên sâu
├── code/         ← Code snippets, config mẫu
├── images/       ← Ảnh, diagram download về local
└── _inbox/       ← Nơi Web Clipper đổ bài mới vào

Tầng 2: Wiki — Sản phẩm của AI

Đây là "cuốn bách khoa" — do AI viết và duy trì 100%. Bạn đọc nó, nhưng không sửa trực tiếp. Mỗi lần có tài liệu mới, AI cập nhật wiki: viết bài mới, sửa bài cũ, tạo liên kết.

wiki/
├── _index.md     ← Mục lục tổng — AI đọc đầu tiên
├── _log.md       ← Nhật ký hoạt động (append-only)
├── _sources.md   ← Danh sách tài liệu đã xử lý
├── topic-a/      ← Folder theo chủ đề
│   ├── _index.md ← Mục lục folder
│   ├── concept-1.md
│   └── concept-2.md
├── topic-b/
└── outputs/      ← Kết quả Q&A, research

Tầng 3: Schema — "Luật chơi" cho AI

Đây là file CLAUDE.md (hoặc AGENTS.md) nằm ở gốc project. Nó dạy AI: cách wiki được tổ chức, quy ước đặt tên, format bài viết, và workflow khi thêm/sửa/tìm kiếm. Bạn và AI cùng cải tiến file này theo thời gian.

Tại sao 3 tầng này hiệu quả?

Raw = sự thật gốc, không ai sửa. Wiki = kiến thức đã tổng hợp, AI duy trì. Schema = luật chơi, bạn và AI cùng cải tiến. Tách biệt 3 tầng → mỗi tầng làm đúng 1 việc → không bao giờ lẫn lộn.

04 Cài đặt — Bắt đầu trong 20 phút

Bạn cần 3 thứ, tất cả miễn phí (trừ API key):

1

Cài Obsidian

Tải từ obsidian.md. Miễn phí cho personal use. Đây là "màn hình" để bạn xem wiki — giống Word nhưng cho markdown, có graph view hiển thị liên kết giữa các bài.

2

Cài Obsidian Web Clipper

Extension trên Chrome/Edge. Khi đọc bài hay trên web → click 1 nút → bài viết được save thành file markdown vào raw/_inbox/. Đây là cách nhanh nhất để thu thập tài liệu.

3

Chuẩn bị LLM Agent

Bạn cần một AI agent có thể đọc/ghi file trên máy. Các lựa chọn phổ biến:

ToolĐặc điểmChi phí
Claude CodeCLI, mạnh, đọc/ghi file trực tiếpTheo API usage
Cursor / WindsurfIDE có AI tích hợp, dễ dùng hơnFree tier có, Pro ~$20/tháng
OpenAI CodexCLI, tương tự Claude CodeTheo API usage
4

Tạo folder project

# Windows PowerShell
mkdir D:\my-wiki
cd D:\my-wiki
mkdir raw\articles, raw\papers, raw\code, raw\images, raw\_inbox
mkdir wiki\outputs
git init

Mở Obsidian → Open vault → chọn D:\my-wiki. Xong — bạn đã có workspace.

5

Tạo CLAUDE.md — "Luật chơi"

Tạo file CLAUDE.md ở gốc project. Copy template dưới đây, sửa phần [CHỦ ĐỀ] thành domain của bạn:

# Wiki Operating Manual

## Identity
Bạn là người duy trì knowledge base về [CHỦ ĐỀ].
Bạn biên dịch tài liệu thô thành wiki có cấu trúc.

## Quy tắc
- raw/ là bất biến — KHÔNG BAO GIỜ sửa file trong raw/
- wiki/ là workspace của bạn — bạn sở hữu mọi file ở đây
- Đọc wiki/_index.md TRƯỚC mọi thao tác
- Ghi log vào wiki/_log.md SAU mọi thao tác
- Mỗi bài viết phải có: TLDR, nội dung, nguồn trích dẫn
- Dùng [[wikilinks]] để liên kết giữa các bài
Xong!

5 bước trên mất khoảng 20 phút. Bạn đã có workspace sẵn sàng. Phần còn lại là học 4 thao tác: Thu thập → Biên dịch → Hỏi-đáp → Bảo trì.

05 Thu thập — Đổ "nguyên liệu" vào

Đây là bước duy nhất bạn phải làm thủ công. Nhưng nó rất nhanh — mỗi lần chỉ vài giây.

Cách 1: Web Clipper (phổ biến nhất)

Đang đọc bài hay trên blog → click icon Web Clipper → chọn save vào raw/_inbox → xong. Bài viết được chuyển thành markdown tự động.

Cách 2: Copy-paste thủ công

Tạo file .md mới trong raw/articles/, paste nội dung vào. Phù hợp cho ghi chú cá nhân, meeting notes, hoặc nội dung không có trên web.

Cách 3: Download file

PDF, code repo, dataset — copy vào raw/ folder phù hợp.

Mẹo quan trọng: Download ảnh về local

Trong Obsidian Settings → Files and links → đặt Attachment folder path = raw/images/. Sau khi clip bài, bấm Ctrl+Shift+D để download ảnh về local. Lý do: link ảnh trên web có thể hỏng bất cứ lúc nào, ảnh local thì luôn còn.

Không cần hoàn hảo

Đừng suy nghĩ quá nhiều "bài này thuộc folder nào". Cứ dump hết vào raw/_inbox/. AI sẽ phân loại khi biên dịch. Việc của bạn chỉ là thu thập — đừng để khâu tổ chức làm bạn ngại thu thập.

06 Biên dịch — AI chuyển nguyên liệu thành kiến thức

Đây là bước "phép thuật". Bạn bảo AI đọc tài liệu thô và cập nhật wiki. Cụ thể:

1

Bạn ra lệnh

# Ví dụ dùng Claude Code
claude "Đọc CLAUDE.md, rồi biên dịch file raw/_inbox/bai-viet-moi.md vào wiki"
2

AI đọc và phân tích

AI đọc tài liệu mới → đọc wiki/_index.md để hiểu cấu trúc hiện tại → quyết định:

• Tạo bài mới? (Nếu chủ đề chưa có trong wiki)

• Bổ sung bài cũ? (Nếu chủ đề đã tồn tại → merge thông tin mới vào)

• Tạo liên kết? (Nếu bài mới liên quan đến bài đã có)

3

AI viết/cập nhật wiki

AI tạo hoặc sửa file .md trong wiki/. Mỗi bài viết có format chuẩn:

---
title: Tên bài viết
tldr: "Tóm tắt 1-2 câu để scan nhanh"
sources:
  - raw/articles/bai-viet-moi.md
tags: [tag1, tag2]
last_compiled: 2026-04-05
---

## TLDR
Tóm tắt ngắn gọn nội dung chính...

## Nội dung
Nội dung chi tiết với [[liên kết]] đến bài khác...

## Nguồn
- Tên bài gốc: [raw/articles/bai-viet-moi.md]
4

AI cập nhật index và log

AI thêm bài mới vào _index.md, ghi nhật ký vào _log.md, đánh dấu source đã xử lý trong _sources.md.

5

Bạn review trong Obsidian

Mở Obsidian → xem bài mới → kiểm tra liên kết có đúng không. Nếu OK → git commit. Nếu không OK → bảo AI sửa lại hoặc git revert.

Một tài liệu có thể "chạm" 10-15 bài wiki

Một bài viết về JPA performance có thể: tạo 1 bài mới về N+1 query, cập nhật bài về connection pooling, thêm liên kết vào bài về testing, bổ sung vào glossary. Đây chính là sức mạnh của wiki — kiến thức được tích hợp, không chỉ được lưu trữ.

07 Hỏi-đáp — Khai thác wiki

Khi wiki đủ lớn (từ ~20 bài trở lên), bạn bắt đầu thấy sức mạnh: hỏi câu hỏi phức tạp mà AI trả lời từ kiến thức đã tổng hợp, không phải từ internet chung chung.

# Hỏi trực tiếp
claude "Đọc wiki/_index.md, tìm bài liên quan, rồi trả lời:
So sánh 3 cách fix N+1 query, cách nào phù hợp nhất cho project có 50+ entities?"

# Xuất ra file markdown
claude "Tổng hợp checklist performance cho JPA, lưu vào wiki/outputs/jpa-checklist.md"

# Xuất ra slide
claude "Tạo slide tóm tắt kiến thức về Spring Security OAuth2, format Marp"

Cách AI tìm câu trả lời

AI không đọc toàn bộ wiki (tốn tokens). Quy trình thông minh hơn:

Đọc _index.md

→
Chọn bài liên quan

Có trích nguồn
Vòng lặp tích lũy — Bí quyết quan trọng nhất

Khi AI trả lời một câu hỏi hay — ví dụ so sánh 3 phương pháp — kết quả đó có giá trị. Đừng để nó biến mất trong lịch sử chat. Hãy "file" kết quả ngược vào wiki. Lần sau hỏi câu tương tự, wiki đã có sẵn câu trả lời — AI không cần phân tích lại. Kiến thức tích lũy giống lãi kép: mỗi câu hỏi làm wiki giàu hơn.

08 Bảo trì — Giữ wiki "khỏe mạnh"

Wiki cũng như vườn cây — cần chăm sóc định kỳ. Tin tốt: AI làm hầu hết việc này cho bạn.

claude "Chạy health check trên wiki. Tìm:
1. Mâu thuẫn giữa các bài (bài A nói X, bài B nói Y)
2. Thông tin cũ cần cập nhật
3. Bài 'mồ côi' không có liên kết từ bài nào khác
4. Khái niệm quan trọng được nhắc đến nhưng chưa có bài riêng
5. Liên kết bị thiếu giữa các bài liên quan
Ghi kết quả vào wiki/outputs/lint-report.md"

Một phần quan trọng của lint: AI đọc wiki và gợi ý "bạn nên tìm hiểu thêm về X" hoặc "có vẻ thiếu thông tin về Y — nên tìm source bổ sung". Đây là cách wiki "tự biết" mình thiếu gì.

09 Quy trình hàng ngày — 15 phút/ngày

Nguyên tắc vàng

Nếu quy trình mất hơn 15 phút/ngày, bạn sẽ bỏ cuộc. Giữ nó nhẹ nhàng. Wiki lớn lên từ từ — không cần vội.

Mỗi ngày — 5 phút sáng

Thu thập. Đọc feed bình thường. Gặp bài hay → Web Clipper → raw/_inbox/. Không cần suy nghĩ, không cần phân loại. Chỉ clip và đi.

Mỗi ngày — 10 phút tối

Biên dịch. Chạy compile cho những gì mới thu thập trong ngày. Review nhanh trong Obsidian — liên kết có đúng không, TLDR có chính xác không. Commit.

Khi cần — Bất kỳ lúc nào

Hỏi-đáp. Gặp vấn đề trong công việc → hỏi wiki trước, hỏi Google sau. Nếu wiki trả lời tốt → bạn tiết kiệm 30 phút search. Nếu trả lời thiếu → biết wiki cần bổ sung ở đâu.

Cuối tuần — 15 phút

Bảo trì. Chạy lint. Đọc report. Xem wiki đã lớn thêm bao nhiêu trong tuần. Cảm nhận kiến thức tích lũy.

Mỗi tháng — 15 phút

Review schema. Mở CLAUDE.md → conventions nào hoạt động tốt? Cần thêm/bớt gì? Cùng AI cải tiến "luật chơi".

10 Mẹo hay từ thực tế

Về Obsidian

Graph View là cách tốt nhất để "nhìn" wiki. Những bài có nhiều liên kết = khái niệm quan trọng. Những bài không có liên kết = có thể cần bổ sung hoặc xóa.

Dataview plugin cho phép query wiki metadata. Ví dụ: tạo bảng tự động liệt kê "tất cả bài có confidence = low" hoặc "bài chưa cập nhật 30 ngày".

Marp plugin cho phép xem slide trực tiếp trong Obsidian. Rất hữu ích khi AI tạo slide từ wiki content.

Về tổ chức

TLDR ở đầu mỗi bài giúp cả bạn và AI scan nhanh. AI đọc index → đọc TLDR → chỉ đọc full bài khi TLDR relevant. Tiết kiệm tokens đáng kể khi wiki lớn.

Git cho wiki = bạn có version history miễn phí. Sai thì revert. Branch thử nghiệm. Push lên GitHub để backup. Wiki chỉ là folder markdown — git xử lý hoàn hảo.

Đừng phân loại quá sớm. Bắt đầu với 3-4 folders. Khi wiki lớn lên, bạn sẽ thấy tự nhiên cần tách folder nào. Để AI gợi ý restructure khi lint.

Về chất lượng

Tách sự thật và suy luận. Khi AI viết wiki, yêu cầu nó đánh dấu rõ: đâu là facts từ source, đâu là inference do AI tổng hợp. Ví dụ: "(inferred)" sau những nhận định AI rút ra nhưng source không nói trực tiếp.

Source attribution. Mỗi claim quan trọng trong wiki nên link về source gốc: "HikariCP default pool size = 10 [source: raw/articles/hikari-tuning.md]". Khi bạn nghi ngờ → click vào source → đọc gốc để verify.

Confidence levels. Đánh dấu mỗi bài là high/medium/low confidence. High = nhiều source corroborate. Low = chỉ 1 source hoặc AI infer. Giúp bạn biết nên tin bao nhiêu.

Về search khi wiki lớn

Ở quy mô nhỏ (~100 bài), file _index.md đủ dùng — AI đọc index rồi navigate. Khi wiki vượt 200+ bài, cân nhắc dùng qmd (github.com/tobi/qmd) — search engine local cho markdown với BM25 + vector search, chạy hoàn toàn trên máy.

11 Câu hỏi thường gặp

Tôi cần biết code không?

Không bắt buộc. Nếu dùng Cursor hoặc Windsurf, mọi thao tác đều qua chat. Nếu dùng Claude Code, cần biết mở terminal và gõ lệnh cơ bản. Không cần biết lập trình.

Tốn bao nhiêu tiền?

Obsidian: miễn phí. Git: miễn phí. Chi phí duy nhất là LLM API — khoảng $5-20/tháng tùy mức dùng. Compile 1 bài tốn ~$0.02-0.05. Nếu compile 2 bài/ngày → ~$3/tháng.

Wiki có thể lớn đến đâu?

Karpathy chạy ~100 bài, ~400K từ mà không cần RAG phức tạp. Ở quy mô ~200-500 bài, thêm search engine (qmd) là đủ. Vượt 1000 bài → cân nhắc finetune model để "biết" wiki trong weights thay vì chỉ qua context.

Nếu AI viết sai thì sao?

3 lớp bảo vệ: (1) Source attribution — mỗi claim link về nguồn gốc, bạn verify được. (2) Confidence levels — bài "low confidence" bạn biết cần cẩn thận. (3) Git — sai thì revert. Quan trọng: đừng tin wiki 100% — verify claim quan trọng bằng source gốc.

Dùng cho topic gì được?

Bất kỳ lĩnh vực nào bạn tích lũy kiến thức theo thời gian. Ví dụ: lập trình (Spring Boot, Python), nghiên cứu (AI, blockchain), business (market analysis), cá nhân (health, finance), sở thích (wine, photography), và nhiều hơn nữa. Một người có thể chạy nhiều KB cho nhiều topic khác nhau.

Khác gì Notion / Confluence / wiki thường?

Sự khác biệt lớn nhất: ai duy trì nó. Wiki thường do con người viết và duy trì → con người bận → wiki chết. LLM wiki do AI duy trì → chi phí maintain gần bằng 0 → wiki sống và lớn lên liên tục. Con người bỏ cuộc với wiki vì gánh nặng bảo trì tăng nhanh hơn giá trị. LLM giải quyết vấn đề này.

Tôi nên bắt đầu từ đâu?

Chọn 1 topic bạn đang thật sự cần hàng ngày. Cài đặt theo Section 04. Clip 5 bài đầu tiên. Compile. Hỏi 1 câu. Nếu câu trả lời hữu ích → bạn sẽ tự muốn tiếp tục. Nếu không → thử topic khác.
